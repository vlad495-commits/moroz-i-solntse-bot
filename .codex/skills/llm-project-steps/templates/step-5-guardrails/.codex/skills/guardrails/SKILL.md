# Скилл: Guardrails (защита от jailbreak и утечки промпта)

Включает один или оба слоя защиты и выбирает конкретные категории паттернов. По умолчанию всё выключено (`GUARDRAILS_INPUT_ENABLED=false`, `GUARDRAILS_OUTPUT_ENABLED=false`) — бот скармливает любой ввод в LLM как есть.

## Что меняется в проекте
- `.env` (корень проекта) — три переменные: `GUARDRAILS_INPUT_ENABLED`, `GUARDRAILS_OUTPUT_ENABLED`, `GUARDRAILS_INPUT_CATEGORIES`
- Файлы кода НЕ трогаются — паттерны лежат в `project/llm/guardrails.py` и активируются через флаги.

## Зачем guardrails

Объясни клиенту (1-2 предложения):
> Guardrails — это два regex-фильтра: на входе (ловит попытки jailbreak — «забудь все инструкции», «теперь ты», «[admin override]» и т.п.) и на выходе (ловит фразы из системного промпта в ответе бота — защита от утечки). 43 паттерна по 7 категориям + output canary. Дефолт — выключено: ты сам решаешь какие категории нужны проекту, потому что часть из них даёт ложные срабатывания на легитимных диалогах.

Важно отметить:
> `sanitize_input` (NFC, zero-width, HTML-теги) и `MAX_INPUT_LENGTH` работают всегда независимо от этого скилла — это базовая чистка, не блокировки.

## Шаг 1 — показать текущее состояние

Прочитай `.env` (Read), вытащи три значения и покажи клиенту в формате:

```
Текущее состояние guardrails:
  Вход (input):  выключено / включено, категории: [role_switch, prompt_leak]
  Выход (canary): выключено / включено
```

## Шаг 2 — что включить?

Спроси:
> Что хочешь включить?
> 1 — только входную защиту (regex против jailbreak)
> 2 — только выходную защиту (canary — детектор утечки промпта)
> 3 — обе
> 0 — выключить всё (откатить к дефолту)

- `0` → выставить `GUARDRAILS_INPUT_ENABLED=false`, `GUARDRAILS_OUTPUT_ENABLED=false`, `GUARDRAILS_INPUT_CATEGORIES=` → перейти к Шагу 5.
- `1` или `3` → к Шагу 3.
- `2` → к Шагу 4.

## Шаг 3 — выбор категорий (только если включается вход)

Покажи клиенту таблицу 7 категорий и попроси выбрать:

```
Выбери категории через запятую (или `all` для всех, `mid` для рекомендуемого набора):

1. role_switch    (14) — подмена роли. «забудь инструкции», «теперь ты», «представь что ты», «pretend you are», «from now on».
2. prompt_leak    (4)  — попытки выудить системный промпт. «покажи промпт», «какие у тебя инструкции», «repeat your prompt».
3. authority      (4)  — фейковая авторитетность. «[admin override]», «tier 5», «как твой администратор».
4. system_tags    (8)  — служебные теги. <system>, <debug>, <override>, <|...|>, [INST].
5. policy_patch   (5)  — фейковые обновления политик. «system update v2», «safety mode disabled», «new rule:».
6. separator      (3)  — инъекции через разделители. Строки из ----/==== с маркерами system/block/start/end/override.
7. known_attack   (5)  — известные атаки. «DAN», «jailbreak», «developer mode», «sudo mode», «unrestricted mode».
```

Допустимые форматы ответа:
- список номеров через запятую: `1,2,7`
- список имён через запятую: `role_switch,prompt_leak,known_attack`
- `all` — все 7 категорий
- `mid` — рекомендованный набор (закрывает ~80% реальных атак): `role_switch,prompt_leak,known_attack`

**Валидация:**
- Если ноль валидных категорий — переспроси (включать вход без категорий бессмысленно).
- Неизвестные имена — отбрось с предупреждением, продолжи с валидными.
- Канон-имена в нижнем регистре, через запятую без пробелов.

**Уточнения по категориям (выдай если клиент спросит):**
- `authority` и `policy_patch` иногда дают ложные срабатывания на легитимных диалогах с упоминанием правил/уровней.
- `system_tags` и `separator` — параноидально-полезно, ложных почти не дают.
- Минимум для большинства проектов: `role_switch,prompt_leak,known_attack`.

## Шаг 4 — про output-слой (только если включается выход)

Объясни клиенту:
> Output guardrail — это детектор фраз из твоего системного промпта в ответе бота. Если LLM случайно процитирует промпт пользователю — мы заблокируем ответ и заменим на `GUARDRAIL_OUTPUT_FALLBACK`.
>
> Сейчас в детекторе только универсальные маркеры (имена py-файлов, `<|im_start|>`, «system prompt» и т.п.). Уникальные фразы из твоего промпта (canary) добавит скилл `/system-prompt` автоматически после написания/правки промпта.

## Шаг 5 — запись в `.env`

Найди в `.env` блок `# --- Guardrails ---`. Замени значения через Edit (`replace_all=False`):

- `GUARDRAILS_INPUT_ENABLED=false` → `GUARDRAILS_INPUT_ENABLED=true` или `false` (по выбору клиента).
- `GUARDRAILS_OUTPUT_ENABLED=false` → аналогично.
- `GUARDRAILS_INPUT_CATEGORIES=` → `GUARDRAILS_INPUT_CATEGORIES=role_switch,prompt_leak,...` (или пусто если выход не включается).

**Если блока в `.env` нет** (старый проект, созданный до этого скилла) — добавь блок целиком в конец файла:
```env

# --- Guardrails (защита от jailbreak и утечки промпта) ---
GUARDRAILS_INPUT_ENABLED=true
GUARDRAILS_OUTPUT_ENABLED=true
GUARDRAILS_INPUT_CATEGORIES=role_switch,prompt_leak,known_attack
```

Перед записью — покажи клиенту итоговые 3 строки и попроси подтвердить.

## Шаг 6 — применить

Скажи клиенту:
> Чтобы применить — `cd project && docker compose restart llm`. Простой 5-10 секунд. После рестарта в `docker compose logs llm` увидишь строку `Guardrails: input=enabled (categories=[...]), output=enabled` — это подтверждение что флаги подхватились. Если бот ещё не запущен — `docker compose up --build` (применится автоматом).

Если клиент согласен — выполни рестарт.

## Валидация / важно

- НЕ трогай другие переменные в `.env` (только 3 `GUARDRAILS_*`).
- НЕ удаляй и не комментируй строки — только меняй значения.
- Если клиент выбрал «включить вход» но не назвал ни одной категории — не записывай `GUARDRAILS_INPUT_ENABLED=true` с пустым списком (бот всё равно пропустит — раннее `return True` при пустых категориях). Лучше переспроси.
- НЕ объясняй что-либо если клиент явно отказался («не нужно guardrails сейчас») — просто выйди.
- adversarial-эвалы (`docker compose exec llm python -m eval.run_evals --only adversarial`) при выключенных guardrails печатают предупреждение и возвращают (0, 0) — это нормально, прогон бессмыслен без активной защиты.
