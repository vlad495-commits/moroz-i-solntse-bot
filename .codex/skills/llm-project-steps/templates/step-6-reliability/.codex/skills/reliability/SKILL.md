# Скилл: Надёжность LLM

Настраивает 3 группы параметров: резервная модель (fallback), количество retry, ID Telegram-чата для алертов.

## Что меняется в проекте
- `.env` — `RESERVE_API_KEY`, `RESERVE_MODEL`, `RESERVE_BASE_URL`, `LLM_MAX_RETRIES`, `ADMIN_TG_CHAT_ID`

## Структура диалога

Скажи клиенту:
> Настроим надёжность LLM. Пройдёмся по 3 пунктам — можно пропустить любой.

### 1. Резервная LLM-модель (страховка от падения основной)

> Когда основной провайдер падает (раз в месяц-два бывает), бот переключается на резервный. Это страховка от downtime.
>
> Поддерживается **любой** провайдер: OpenAI, Anthropic, OpenRouter, DeepSeek, Together AI, Groq, локальный Ollama и т.д.
> Стандартная пара — резерв у другого провайдера чем основной (если упал OpenAI — переключаемся на Anthropic, и наоборот).
>
> Какую модель использовать как резервную? (введи название или «пропустить»)

Если «пропустить» — оставь `RESERVE_*` пустыми.

Иначе:
> Введи API-ключ для резервной модели:

Запиши в `RESERVE_API_KEY`.

**Определение `RESERVE_BASE_URL`:**
- OpenAI / Anthropic native → `RESERVE_BASE_URL=` (пусто)
- OpenRouter → `https://openrouter.ai/api/v1`
- DeepSeek → `https://api.deepseek.com`
- Together AI → `https://api.together.xyz/v1`
- Groq → `https://api.groq.com/openai/v1`
- Mistral → `https://api.mistral.ai/v1`
- Ollama → `http://host.docker.internal:11434/v1`
- Незнакомое → делегируй research-агенту по шаблону «Найди endpoint и совместимость для модели X».

Запиши `RESERVE_MODEL=`, `RESERVE_API_KEY=`, `RESERVE_BASE_URL=` в `.env`.

### 2. LLM_MAX_RETRIES — количество попыток

> Сколько повторов делать на основном LLM при ошибке прежде чем переключиться на резервный.
> Повтор через паузу с exponential backoff (2^attempt сек): 1-й повтор — через 1 сек.
>
> Дефолт: **1** (= 2 попытки итого: первая + 1 retry).
> Каждая попытка ограничена `LLM_REQUEST_TIMEOUT_SEC`. Если две попытки подряд упали —
> третья почти наверняка тоже упадёт, быстрее уйти на резерв.
> Увеличивать (2-3) — если резерв дорогой и хочешь сильнее держаться за основной.
>
> Твоё значение [1]:

Запиши в `LLM_MAX_RETRIES`.

### 3. ADMIN_TG_CHAT_ID — алерты администратору

> ID Telegram-чата куда бот шлёт алерты при ошибках (CRITICAL/WARNING). Без этого алерты не работают.
>
> Как получить: напиши в Telegram боту **@userinfobot** — он пришлёт твой `id`. Это число (например, `123456789`).
> Если хочешь алерты в групповой чат — добавь @userinfobot в группу, он покажет ID группы (отрицательное число).
>
> Введи ID для алертов (или «пропустить»):

Валидируй: целое число (может быть отрицательным для групп). Запиши в `ADMIN_TG_CHAT_ID`.

## Применение

После всех изменений:
> Применить — `cd project && docker compose restart llm`. ~10 секунд простоя.

## Важно

- НЕ перезаписывай `.env` целиком — меняй только нужные ключи. Сохрани комментарии и форматирование.
- Если клиент пропускает пункт — не настаивай.
- Не настраивай ADMIN_PASSWORD, ADMIN_USERNAME — это в скилле `/production-ready`.
