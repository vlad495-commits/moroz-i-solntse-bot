# Скилл: Production-ready (финальный аудит)

Прогоняет финальный чек-лист готовности перед сдачей клиенту. Сменяет дефолтные пароли админки, проверяет ключевые настройки `.env`.

**Что НЕ настраивает (см. отдельные скиллы):**
- Резервная LLM, retry, ADMIN_TG_CHAT_ID → `/reliability`
- Hardening сервера → `/server-hardening`
- SSH → `/ssh-setup`
- Деплой → `/deploy`

## Что меняется в проекте
- `.env` — `ADMIN_USERNAME`, `ADMIN_PASSWORD`, `ADMIN_SESSION_SECRET`

## Структура диалога

Скажи клиенту:
> Финальный аудит перед сдачей. Сменим дефолтные пароли и пройдёмся по чек-листу готовности.

### 1. Смена дефолтных учётных данных админки

Прочитай текущие значения `ADMIN_USERNAME` и `ADMIN_PASSWORD` в `.env`.

> Сейчас логин/пароль админки: `<ADMIN_USERNAME>` / `<ADMIN_PASSWORD>`.
>
> Если это `admin` / `admin` — **обязательно** поменять перед деплоем (любой кто узнал URL админки получает доступ).
>
> Какой логин ставим? (Enter = оставить как есть)
> Какой пароль? (минимум 16 символов, лучше через генератор `openssl rand -base64 24`)

Запиши новые значения в `ADMIN_USERNAME`, `ADMIN_PASSWORD`. Если оставлены пустыми — не трогай.

### 2. ADMIN_SESSION_SECRET

Проверь:
```bash
grep "^ADMIN_SESSION_SECRET=" <root>/.env
```

Если значение `change-me-in-production` или пустое — сгенерируй и запиши:
```bash
NEW_SECRET=$(openssl rand -hex 32)
sed -i '' "s|^ADMIN_SESSION_SECRET=.*|ADMIN_SESSION_SECRET=${NEW_SECRET}|" <root>/.env
```

### 3. Финальный чек-лист

Прочитай `.env`, заполни галочки:

```
Production-ready чек-лист:

LLM
[✓/✗] LLM_API_KEY заполнен
[✓/✗] LLM_MODEL: <значение>
[✓/✗] LLM_BASE_URL: <значение или "default">
[✓/✗] RESERVE_MODEL: <значение или "не настроен">  (опц., рекомендую — настрой через /reliability)

Алерты
[✓/✗] ADMIN_TG_CHAT_ID заполнен  (настроить через /reliability)
[✓/✗] TELEGRAM_BOT_TOKEN заполнен

Безопасность
[✓/✗] ADMIN_USERNAME изменён с дефолта
[✓/✗] ADMIN_PASSWORD не "admin"
[✓/✗] ADMIN_SESSION_SECRET не "change-me-in-production"
[✓/✗] POSTGRES_PASSWORD не дефолтный (длинный hex)
[✓/✗] REDIS_PASSWORD задан

Промпт и эвалы
[✓/✗] system.md не пустой  (или используется без промпта осознанно)
[✓/✗] dataset.json не пустой  (минимум 10 кейсов рекомендуется)

Сервер (если деплоен)
[ ] UFW настроен (см. /server-hardening)
[ ] fail2ban установлен
[ ] SSH-ключи настроены, парольный логин выключен
[ ] postgres-backup сервис в docker-compose.yml
[ ] Off-site копия бэкапа настроена (облако / вторая машина)
[ ] Восстановление из бэкапа проверено (развёрнут тестовый дамп)
[ ] Caddy/TLS настроен (если нужен публичный домен)
```

Заполни галочки на основе фактического состояния `.env` и наличия сервиса `postgres-backup` в `docker-compose.yml`.

Если есть НЕзаполненные критичные пункты — выдели и скажи что дозаполнить (через какой скилл).

Если всё ок:
> Проект готов к сдаче клиенту. Финальные шаги:
> - Запусти /deploy (если ещё не задеплоен)
> - Передай клиенту: URL админки, логин/пароль, имя бота, контакты для поддержки
> - Сделай тестовый прогон эвалов: `docker compose exec bot python -m eval.run_evals` (опц.)

### 4. Применение

```bash
cd project && docker compose restart bot admin
```
~10 секунд простоя.

## Важно

- НЕ перезаписывай `.env` целиком — меняй только нужные ключи. Сохрани комментарии и форматирование.
- Если клиент пропускает пункт — не настаивай. Чек-лист сам подсветит что не сделано.
- Серверные пункты (UFW, fail2ban, бэкапы) проверяй по факту — на локальном проекте они не нужны.
