# Скилл: Эвалы (тестирование бота)

Настраивает 3 параметра эвалюэйшенов и помогает наполнить `dataset.json` тест-кейсами.

## Что меняется в проекте
- `.env` — `JUDGE_MODEL`, `JUDGE_API_KEY`, `JUDGE_PASS_THRESHOLD`
- `project/llm/eval/dataset.json` — тест-кейсы клиента

## Структура диалога

### Часть А — настройка judge (если ещё не настроен)

Проверь `.env`: если `JUDGE_MODEL` и `JUDGE_PASS_THRESHOLD` уже заданы (не дефолты) — пропусти Часть А, спроси «настройка judge не нужна, переходим к кейсам?».

#### А.1 — JUDGE_MODEL
> Эвалы у нас двухступенчатые. Сначала regex-проверка ответа (быстро, бесплатно). Если regex не подтвердил — зовём LLM-судью, который оценивает ответ бота по смыслу.
>
> Какую модель использовать как судью?
> - `gpt-4.1-mini` (дефолт) — дёшево и точно, $0.022 за 50 кейсов
> - `gpt-4.1` — строже, но в 5 раз дороже
> - `gpt-4o` — другой стиль оценки
> - `claude-sonnet-4-6` — через Anthropic
> - другая (введи название)

Запиши ответ в `JUDGE_MODEL` в `.env`.

#### А.2 — JUDGE_API_KEY и JUDGE_BASE_URL
> Если judge у того же провайдера что и основной бот — оставим пустым, будет использован `LLM_API_KEY` и `LLM_BASE_URL`.
> Если judge у другого провайдера (например, ты выбрал `claude-sonnet-4-6` как судью, а бот на OpenAI) — нужен отдельный ключ.
>
> Введи API-ключ для judge или оставь пусто:

Запиши в `JUDGE_API_KEY` (или оставь пустой).

Если judge — у нестандартного провайдера (не OpenAI/Anthropic native), определи `JUDGE_BASE_URL` по той же логике что и для основной LLM (см. таблицу провайдеров в скилле `client-project`). Если незнакомый — делегируй research-агенту. Запиши в `JUDGE_BASE_URL`.

#### А.3 — JUDGE_PASS_THRESHOLD
> Judge выдаёт оценку от 0.0 до 1.0. Если score >= порога — кейс прошёл, иначе провалился.
>
> Дефолт: **0.8** (строго — близко к идеалу)
> 0.7 — мягче, больше PASS
> 0.9 — очень строго, для критичных проектов (медицина, финансы)
>
> Твоё значение [0.8]:

Запиши в `JUDGE_PASS_THRESHOLD`.

После А — скажи: «Настройки записаны. Применятся при следующем рестарте админки. Теперь — кейсы.»

### Часть Б — наполнение dataset.json

Спроси клиента:
> Как добавляем тест-кейсы?
> 1. Я хочу добавить конкретный кейс (вопрос → ожидаемый ответ)
> 2. Помоги сформировать кейсы из реальной истории чатов

#### Режим 1 — добавление конкретного кейса

Цикл, пока клиент не скажет «хватит / достаточно»:

1. Спроси: **«Какой вопрос должен задавать пользователь?»**
2. Спроси: **«Какие ключевые слова ОБЯЗАТЕЛЬНО должны быть в ответе бота?»** (через запятую, либо префикс `r:` для regex). Объясни:
   > Если все эти слова найдены в ответе — кейс автоматически проходит без вызова judge (экономим). Например: `300, рублей` → ответ должен содержать и «300» и «рублей».
3. Спроси: **«Какие слова НЕ должны быть в ответе?»** (forbidden_keywords, можно пусто). Например: `бесплатно, скидка` для бота где этих слов быть не должно.
4. Спроси: **«Какой эталонный ответ ожидаешь?»** — это для LLM-судьи, чтобы он знал с чем сравнивать по смыслу.
5. Покажи итог:
   ```
   Вопрос: ...
   Ожидаемые: [...]
   Запрещённые: [...]
   Эталон: ...
   ```
   Спроси: **«Записать?»**
6. Если да — открой `project/llm/eval/dataset.json`, прочитай как массив объектов, добавь новый объект:
   ```json
   {
     "question": "...",
     "expected_keywords": [...],
     "forbidden_keywords": [...],
     "expected_answer": "..."
   }
   ```
   Сохрани с `indent=2, ensure_ascii=False`.
7. Спроси: **«Ещё кейс?»** — если да, на пункт 1; если нет — выйди.

#### Режим 2 — генерация из истории чатов

1. Спроси: **«Сколько последних реальных вопросов взять как кандидаты? (по умолчанию 30)»**
2. Подключись к Postgres через docker и вытащи реальные user-вопросы:
   ```bash
   docker compose exec -T postgres psql -U "$POSTGRES_USER" -d "$POSTGRES_DB" -t -A -F'|' -c "
     SELECT m1.id, m1.content, m2.content
     FROM messages m1
     LEFT JOIN LATERAL (
       SELECT content FROM messages
       WHERE chat_id = m1.chat_id AND role = 'assistant' AND created_at > m1.created_at
       ORDER BY created_at ASC LIMIT 1
     ) m2 ON true
     WHERE m1.role = 'user'
     ORDER BY m1.created_at DESC
     LIMIT N;
   "
   ```
   (замени N на ответ клиента)
3. Если результат пустой — скажи: «В БД ещё нет истории сообщений. Запусти бота, наберите сообщений с тестовыми пользователями и вернись к этому скиллу.» Выйди.
4. Если есть результаты — для каждого по очереди показывай:
   ```
   Вопрос: <текст>
   Реальный ответ бота: <текст>

   Превратить в тест-кейс?
   1. Да, оставить эталоном этот ответ бота
   2. Да, но я введу другой эталонный ответ
   3. Пропустить (не добавлять)
   4. Хватит, выходим
   ```
5. По выбору:
   - 1 — добавить в dataset.json с этим ответом как `expected_answer`. Спроси про `expected_keywords` (краткий список).
   - 2 — спроси клиента про эталонный ответ + keywords, добавь.
   - 3 — следующий кандидат.
   - 4 — выйди из цикла.
6. После цикла сообщи: «Добавлено N кейсов. Прогнать сейчас? (`docker compose exec admin python -c "..."` или через UI `/eval/`)».

## Важно

- НЕ перезаписывай dataset.json — добавляй к существующему массиву.
- Если файл не валидный JSON — спроси клиента: «Файл повреждён, перезаписать пустым массивом?»
- Адверсариальный датасет (`adversarial_dataset.json`) — НЕ трогай, он универсальный для всех проектов.
- При записи в `.env` сохрани остальные строки и комментарии — меняй только нужные ключи.
