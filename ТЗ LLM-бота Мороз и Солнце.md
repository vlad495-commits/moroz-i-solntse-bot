# ТЗ LLM-бота "Мороз и Солнце"

**Версия:** 0.1  
**Дата:** 2026-07-07  
**Статус:** архитектурное ТЗ для целевой системы и визуальной схемы. Не финальный production-документ.  
**Основа:** текущий проект, системный промпт v1.4, клиентские материалы, ответы пользователя на уточняющие вопросы, конспект урока по архитектуре LLM-ботов.

**Правило дальнейшей работы:** все последующие архитектурные и продуктовые решения по боту сверять с этим ТЗ. Если новое решение противоречит ТЗ, сначала обновить ТЗ, дорожную карту и changelog, затем переходить к реализации.

---

## 1. Цель системы

Система должна стать не просто Telegram-ботом с LLM, а полноценным AI-оператором центра "Мороз и Солнце".

Главные задачи:
- отвечать клиентам по услугам, ценам, подготовке, сертификатам, депозитам, курсам и записи;
- помогать подобрать услугу без медицинских обещаний и персональных рекомендаций;
- записывать клиента через YCLIENTS: смотреть слоты, создавать запись, переносить и отменять;
- предлагать оплату через ЮKassa как опциональный следующий шаг;
- поддерживать follow-up, реактивацию и рассылки;
- передавать сложные вопросы администратору;
- давать владельцу админку для управления знаниями, ценами, диалогами, рассылками, статусами и аналитикой;
- работать предсказуемо: через роутер, сценарии, проверки, эвалы, логи и мониторинг.

Ключевой принцип из урока: сначала подробное ТЗ и архитектура, потом код. Система строится как набор детерминированных сценариев и инструментов, а не как автономный агент "на все случаи".

## 2. Каналы

В целевой архитектуре нужно показать 4 канала:

1. Telegram.
2. WhatsApp.
3. Instagram.
4. ВКонтакте.

Порядок каналов в схеме не важен. Все каналы должны приходить в единый слой адаптеров.

### 2.1. Единый формат входящего сообщения

Каждый канал имеет свой API, ограничения и формат событий. Поэтому снаружи система работает с разными каналами, а внутри приводит все к единому объекту `IncomingMessage`.

Пример внутреннего формата:
- `channel`: telegram / whatsapp / instagram / vk;
- `external_user_id`;
- `chat_id`;
- `message_id`;
- `message_type`: text / audio;
- `text`;
- `audio_url`, если есть;
- `received_at`;
- `metadata`.

### 2.2. Типы сообщений

На первом целевом этапе принимаем:
- текст;
- голосовые/аудио.

Фото, видео, кружки и файлы пока не анализируем. Для них бот должен попросить написать текстом или передать вопрос администратору на будущих этапах.

### 2.3. Голосовые сообщения

Голосовые сообщения транскрибируются через Yandex SpeechKit.

Решение по хранению:
- оригинальные аудиофайлы не хранить постоянно;
- хранить транскрибацию, статус распознавания, технические метаданные;
- оригинал можно хранить временно до 24 часов для отладки ошибок, если это нужно и юридически допустимо.

Если транскрибация не удалась, бот просит клиента написать вопрос текстом. При повторной проблеме можно создать эскалацию администратору.

## 3. Общий поток сообщения

```text
Клиент в канале
-> адаптер канала
-> нормализация сообщения
-> транскрибация, если аудио
-> privacy gate: проверка согласия на обработку ПД
-> буфер сообщений 5 секунд
-> валидация входа
-> security / guardrails
-> маскирование ПД перед внешней LLM
-> сбор контекста и базы знаний
-> роутер намерений
-> один или несколько сценариев
-> инструменты: YCLIENTS / ЮKassa / БД / scheduler / админка
-> LLM-ответ или шаблонный ответ
-> output security и валидатор
-> восстановление разрешенных ПД из текущей сессии
-> ответ клиенту
-> журнал событий, метрики, история
```

До получения согласия система хранит только минимальные технические данные, необходимые для показа формы согласия и защиты от злоупотреблений. Содержимое сообщения не должно уходить во внешнюю LLM, попадать в долгую историю или использоваться в аналитике диалога. Отказ ведет к нейтральной заглушке без продолжения обработки.

---

## 4. Буфер сообщений

Быстрые сообщения пользователя должны склеиваться в один запрос.

Настройка по умолчанию:
- задержка буфера: 5 секунд;
- если клиент за 5 секунд отправил несколько сообщений, они объединяются;
- в LLM и роутер уходит единый смысловой запрос.

Пример:

```text
Клиент: здравствуйте
Клиент: хочу в солярий
Клиент: сколько стоит
Клиент: и можно сегодня?

Система отправляет в обработку один объединенный запрос:
"Здравствуйте. Хочу в солярий. Сколько стоит? Можно сегодня?"
```

Зачем нужен буфер:
- бот не отвечает на каждую короткую фразу отдельно;
- диалог выглядит естественнее;
- снижается расход токенов;
- роутер лучше видит полный контекст.

## 5. Роутер и сценарии

В основе системы должен быть роутер-классификатор. Он определяет намерение клиента и выбирает один или несколько сценариев.

Роутер может выбрать несколько сценариев, если запрос мультинтентный. Например:

```text
"Сколько стоит криокапсула и можно записаться завтра?"
```

Сценарии:
1. Цена/FAQ по услуге.
2. Запись.

Приоритет обработки мультинтентов:
1. Безопасность, жалоба, конфликт, медицинский риск.
2. Запись, перенос, отмена.
3. Оплата, сертификат, курс, депозит.
4. Консультация и FAQ.
5. Апселл, follow-up, маркетинговые действия.

Если роутер не уверен в намерении, бот не должен выдумывать. Он задает один уточняющий вопрос или создает эскалацию, если запрос рискованный.

## 6. Основные сценарии

### 6.1. FAQ и консультация

Бот отвечает по утвержденной базе знаний:
- услуги;
- цены;
- длительность;
- подготовка;
- противопоказания в безопасной общей форме;
- отличия услуг;
- адрес, график, контакты;
- депозитный счет;
- сертификаты;
- курсы и программы.

Бот не дает персональных медицинских рекомендаций, не ставит диагнозы и не говорит "вам можно" или "вам нельзя" по индивидуальному состоянию.

### 6.2. Подбор услуги

Бот помогает выбрать направление по цели:
- загар;
- восстановление;
- отдых;
- уход за кожей;
- подарок;
- тонус;
- интерес к конкретной процедуре.

Правило: предлагать 2-3 варианта, а не весь каталог.

### 6.3. Запись через YCLIENTS

YCLIENTS является главным источником слотов и записей. Локальная БД хранит копии, историю, статусы, аналитику, связь с чатами и платежами.

Бот должен поддерживать разные порядки диалога:
- клиент сначала называет услугу;
- клиент сначала называет дату/время;
- клиент сначала выбирает мастера;
- клиент просит ближайшее свободное время;
- клиент хочет несколько услуг за один визит.

Бот предлагает 2-3 ближайших подходящих слота.

Перед созданием записи бот обязательно показывает резюме:
- услуга или список услуг;
- дата;
- время;
- мастер или "любой свободный";
- цена и длительность, если доступны;
- имя клиента;
- телефон;
- комментарий, если есть.

Запись создается только после подтверждения клиента.

### 6.4. Несколько услуг за один визит

Система должна поддерживать запись на несколько услуг за один визит.

Логика:
1. Собрать список услуг.
2. Проверить совместимость и длительность.
3. Найти подходящие окна в YCLIENTS.
4. Показать 2-3 варианта.
5. Перед созданием записи показать резюме.
6. После подтверждения создать запись или связанный набор записей в YCLIENTS.

Точная техническая реализация зависит от возможностей YCLIENTS API.

### 6.5. Конфликт слота

Перед созданием записи система должна повторно проверить слот.

Если слот только что заняли:
- бот сообщает, что время уже недоступно;
- предлагает 2-3 новых ближайших слота;
- не создает ошибочную запись.

### 6.6. Перенос записи

Оптимальное решение:
- искать активные записи по телефону клиента;
- если найдена одна запись, показать ее и предложить новые слоты;
- если найдено несколько, попросить выбрать нужную;
- если запись не найдена уверенно, передать администратору.

Перед переносом показывать резюме старой и новой записи.

### 6.7. Отмена записи

Бот не отменяет запись молча.

Процесс:
1. Найти активную запись.
2. Показать клиенту, какую запись он хочет отменить.
3. Предупредить о правилах отмены.
4. Попросить подтверждение.
5. После подтверждения отменить через YCLIENTS.
6. Записать событие в локальную БД.

Правило отмены/переноса должно быть настраиваемым в админке. Предварительное значение для схемы: не позднее чем за 3 часа до визита. Финальное значение открыто и должно быть подтверждено клиентом.

### 6.8. Fallback при сбое YCLIENTS

Если YCLIENTS недоступен:
- бот дает ссылку на онлайн-запись;
- создает эскалацию администратору;
- пишет в лог ошибку интеграции;
- при необходимости отправляет технический алерт.

### 6.9. Оплата через ЮKassa

ЮKassa включается в целевую архитектуру как платежный модуль.

Базовый порядок:
1. Сначала создается запись или коммерческая заявка.
2. После этого бот предлагает оплату как опциональный вариант.
3. Клиент может оплатить по ссылке или на месте в салоне.

Для услуг/курсов/сертификатов, где предоплата обязательна, это должно быть отдельным правилом в админке.

Платежный модуль:
- создает платеж/платежную ссылку;
- отправляет ссылку клиенту;
- принимает webhook от ЮKassa;
- обновляет статус оплаты в БД;
- уведомляет администратора при успешной/ошибочной оплате.

### 6.10. Курсы, депозиты и сертификаты

Бот должен поддерживать коммерческие сценарии:
- покупка курса;
- пополнение депозитного счета;
- покупка подарочного сертификата.

Общий процесс:
1. Бот объясняет продукт.
2. Уточняет нужные параметры.
3. Показывает резюме покупки.
4. Создает платеж через ЮKassa.
5. Фиксирует покупку в локальной БД.
6. Уведомляет администратора.
7. Синхронизирует с YCLIENTS, если API позволяет.

Формат выдачи сертификата открыт: автоматическая генерация PDF/картинки или ручное оформление администратором.

### 6.11. Эскалация администратору

Первичный канал эскалации: отдельный Telegram-бот для сотрудников.

В эскалацию отправляются:
- прямой запрос "позовите администратора";
- жалобы и конфликтные ситуации;
- вопросы по оплате и возвратам;
- медицински чувствительные вопросы;
- ошибки записи/оплаты;
- неуверенность LLM или роутера;
- ситуации, где нужна ручная проверка.

Бот не должен резко замолкать. Он сообщает клиенту, что вопрос передан администратору, и дальше отвечает только в безопасных рамках. Для конфликтных и чувствительных случаев чат может переходить в режим "ожидает администратора".

### 6.12. Напоминания

В архитектуре нужен отдельный scheduler.

Он отвечает за:
- напоминания о визите;
- повторные напоминания;
- проверку статусов записей;
- follow-up;
- реактивацию;
- авто-включение бота после эскалации, если это будет нужно.

Конкретные сроки напоминаний остаются открытым вопросом.

### 6.13. Follow-up

Follow-up после визита должен быть отдельным сценарным блоком.

Возможные задачи:
- спросить, как прошло посещение;
- собрать оценку;
- предложить повторную запись;
- предложить подходящую следующую услугу;
- передать негативный отзыв администратору.

Для оценки 1-5 нужно хранить связь с завершенным визитом. Низкая оценка запускает служебную задачу администратору и не должна автоматически публиковаться или превращаться в рекламную коммуникацию. Положительная оценка может обновлять агрегированную статистику услуги/специалиста. Повторный запрос оценки по одному визиту запрещен.

Точные сроки и тексты follow-up остаются открытыми.

### 6.14. Реактивация

Реактивация нужна в целевой архитектуре.

Сценарий:
- scheduler находит клиентов без визитов/диалогов заданный срок;
- проверяет согласие на рассылки;
- формирует кампанию;
- отправляет сообщение через доступный канал;
- пишет результат в аналитику.

У каждой реактивации должен быть уникальный ключ кампании и клиента для защиты от дублей. Если клиент отвечает, сообщение возвращается в общий входной поток как новый диалог. Повторный follow-up допустим только по заранее заданной политике, с лимитом попыток и немедленным прекращением после отказа/команды `стоп`.

Срок "спящего клиента" остается открытым.

### 6.15. Рассылки из админки

Владелец должен запускать ручные рассылки из админки.

Обязательный процесс:
1. Черновик рассылки.
2. Предпросмотр.
3. Тестовое сообщение.
4. Подтверждение запуска.
5. Отправка с лимитами.
6. Статистика и стоп-кнопка.

Нужны лимиты:
- дневной лимит;
- лимит скорости;
- защита от повторной отправки одной кампании;
- остановка кампании при ошибках.

## 7. Админка

Админка должна быть полноценным центром управления, а не только внутренней панелью.

Доступ:
- на ближайшем этапе только владелец;
- роли "ресепшен/админ/владелец" пока не усложняем.

Разделы:
- диалоги;
- карточка клиента;
- эскалации;
- записи и статусы YCLIENTS;
- цены и услуги;
- база знаний;
- промпт и настройки LLM;
- платежи ЮKassa;
- курсы/депозиты/сертификаты;
- follow-up и scheduler-задачи;
- рассылки;
- статистика;
- здоровье системы;
- логи;
- аудит изменений.

### 7.1. База знаний и цены

Владелец должен редактировать цены и услуги через админку.

Оптимальное решение:
- Postgres как рабочее хранилище базы знаний;
- версии изменений для отката;
- экспорт/снапшоты в Markdown или JSON для контроля и резервирования.

Это открытый технический вопрос по реализации, но в архитектуре нужно заложить гибридную модель: удобно редактировать в админке и безопасно откатывать изменения.

### 7.2. Версионирование

Нужно версионировать:
- цены;
- услуги;
- базу знаний;
- промпты;
- настройки сценариев;
- шаблоны рассылок.

Причина: если владелец случайно ошибся, должна быть возможность посмотреть историю и откатить.

### 7.3. Аудит действий

Даже если доступ только у владельца, нужен аудит:
- кто изменил;
- что изменил;
- когда;
- старое значение;
- новое значение.

## 8. Хранилища

### 8.1. PostgreSQL

PostgreSQL хранит:
- пользователей;
- канальные идентификаторы;
- диалоги;
- сообщения;
- транскрибации;
- записи и копии данных YCLIENTS;
- платежи;
- покупки курсов/депозитов/сертификатов;
- согласия;
- базу знаний;
- версии промптов;
- eval-кейсы и прогоны;
- логи событий;
- метрики;
- задачи scheduler;
- аудит изменений.

### 8.2. Redis

Redis хранит:
- короткий контекст диалога;
- буфер сообщений;
- флаги состояния;
- временные блокировки;
- кэш быстрых данных;
- состояние очередей, если будет выбран Redis-based worker.

### 8.3. Локальная БД и YCLIENTS

YCLIENTS - источник правды для реальных слотов и записей.

Локальная БД - источник правды для:
- истории взаимодействий;
- аналитики;
- связки чат -> клиент -> запись;
- платежей;
- статусов эскалации;
- задач follow-up;
- аудита.

## 9. Очереди, worker и scheduler

Для визуальной архитектуры нужно показать отдельный слой:

```text
Task Queue + Workers
```

Через него идут:
- транскрибация аудио;
- тяжелые API-вызовы;
- рассылки;
- follow-up;
- реактивация;
- платежные webhooks;
- retry-задачи;
- фоновые проверки.

Конкретную технологию пока не фиксируем. Возможные варианты: Celery, RQ, Redis Queue или другой worker.

`Scheduler` должен быть отдельным блоком. Он запускает задачи по времени и не должен зависеть от входящих сообщений клиента.

## 10. LLM-архитектура

В архитектуре нужно разделять роли LLM:

1. Роутер-классификатор намерений.
2. Основная LLM для клиентского ответа.
3. Summarizer/compact для сжатия длинной истории.
4. Validator для проверки ответа.
5. Guard/security model для рискованных запросов.
6. LLM-judge для eval-прогонов.

Не обязательно все эти роли сразу реализовывать отдельными моделями, но в целевой схеме они должны быть показаны как логические блоки.

### 10.0. Принцип scripts first

Дешевые и однозначные проверки выполняются до LLM: длина, формат, тип вложения, rate limit, пустой ввод, известные команды и явные стоп-слова. Guard-модель или LLM-классификатор подключаются только для неоднозначных случаев. Это снижает стоимость, задержку и количество внешних передач данных.

### 10.1. Compact истории

Если история диалога становится длинной, система должна сжимать ее отдельной LLM или отдельным summarizer-блоком.

Порог из конспекта: больше 30 сообщений.

---

## 11. Безопасность LLM

Безопасность нужна вокруг каждого LLM-вызова.

Общий pipeline:

```text
валидация
-> input security
-> маскирование ПД
-> сбор промпта
-> LLM
-> output security
-> валидатор ответа
-> постпроцессинг
```

### 11.1. Guardrails

Guardrails проектируются перед каждым LLM-вызовом с учетом роли модели:
- роутер;
- основной ответ;
- summarizer;
- валидатор;
- judge.

Проверки:
- jailbreak;
- попытка раскрыть системный промпт;
- off-topic;
- опасные медицинские запросы;
- персональные данные;
- выдуманные факты;
- канареечный токен;
- запрещенные обещания;
- подозрительные инструкции.

### 11.2. Маскирование ПД

Перед внешней LLM маскируются все персональные данные.

Маскировать:
- телефон;
- ФИО;
- email;
- адреса;
- Telegram/WhatsApp/VK/Instagram ID;
- платежные данные;
- медицинские подробности;
- любые свободно введенные ПД.

Реализация: отдельный privacy/security-блок перед внешней LLM. Конкретные инструменты уточняются при реализации. В конспекте упоминались Presidio + regex.

### 11.3. Output validation

Перед отправкой клиенту ответ проверяется:
- не раскрывает ли внутренние инструкции;
- не содержит ли неразрешенные ПД;
- не обещает ли медицинский результат;
- не выдумывает ли цену, слот, услугу;
- не нарушает ли тон и границы бота.

Если ответ не проходит проверку:
- одна повторная генерация с коротким комментарием валидатора;
- безопасная заглушка;
- эскалация администратору.

Цикл `генерация -> проверка -> исправление` обязан иметь жесткий лимит попыток. После исчерпания лимита система не продолжает тратить токены, а возвращает безопасную заглушку, пишет причину в журнал и при необходимости создает эскалацию.

### 11.4. Восстановление замаскированных данных

Плейсхолдеры ПД можно заменить исходными значениями только после output security и успешной валидации ответа. Разрешено использовать только соответствия из текущей сессии и только те данные, которые нужны для ответа клиенту. Эти соответствия нельзя получать из другой сессии или генерировать самой LLM.

## 12. Согласия и персональные данные

Нужно разделять:
- согласие на обработку ПД;
- согласие на рекламные рассылки;
- сервисные уведомления по записи.

На первом контакте бот должен показывать согласие на обработку ПД. Отдельно нужно запрашивать согласие на рекламные рассылки.

Сервисные уведомления, напоминания и юридические формулировки требуют отдельной проверки перед production.

Срок хранения истории и ПД пока открыт. В текущем проекте по умолчанию фигурирует 3 года, но финальное значение нужно подтвердить перед production.

## 13. Evaluations

Эвалы нужны не только для основного клиентского ответа.

Нужно тестировать отдельно:
- роутер намерений;
- основную LLM;
- guardrails;
- output validator;
- сценарии записи;
- сценарии оплаты;
- fallback и эскалацию.

Eval-раздел пока остается внутренним и завязан на разработчика/владельца проекта. Клиенту-владельцу полный доступ к eval-управлению пока не открываем.

## 14. Надежность

### 14.1. Retry

Нужны ретраи для:
- YCLIENTS;
- ЮKassa;
- LLM;
- Yandex SpeechKit;
- отправки сообщений;
- webhook-обработчиков.

Общая логика:
1. Несколько попыток.
2. Если не удалось - fallback.
3. Если критично - алерт.
4. Если влияет на клиента - честное сообщение и/или эскалация.

### 14.2. Fallback основной LLM

Если основная LLM недоступна:
- попробовать резервную LLM, если настроена;
- если резервной нет или она тоже недоступна - отправить безопасную заглушку;
- создать технический алерт;
- при необходимости передать администратору.

### 14.3. Monitoring & Alerts

Нужно показывать здоровье системы в админке:
- каналы;
- LLM;
- YCLIENTS;
- ЮKassa;
- Yandex SpeechKit;
- Redis;
- Postgres;
- task queue;
- workers;
- scheduler;
- последние ошибки.

Технические алерты отправлять в отдельный Telegram-бот и отображать в админке.

### 14.4. Бэкапы

В целевой архитектуре нужно заложить:
- ежедневные бэкапы Postgres;
- проверку восстановления;
- хранение конфигурации отдельно от секретов;
- план отката.

## 15. Статистика и KPI

Главные KPI:
- количество диалогов;
- количество записей;
- конверсия диалог -> запись;
- количество эскалаций;
- стоимость LLM;
- успешные и неуспешные оплаты;
- ошибки YCLIENTS/ЮKassa/LLM;
- среднее время ответа;
- доля fallback-сценариев.

Эвалы важны как внутренний показатель качества, но в клиентской статистике их можно не выводить на первый план.

### 15.1. Журнал выполнения запроса

Каждый входящий запрос получает `correlation_id`. В событийный журнал по этапам записываются: канал, сценарий, статусы проверок, вызванные инструменты, количество попыток, fallback/эскалация, длительность, токены и стоимость, итоговый статус и `next_action`. Секреты, полный системный промпт и немаскированные ПД в технические логи не попадают.

Сценарии возвращают единый структурированный результат: `status`, `message`, `next_action`, `events`, `cost`, `error_code`. Это позволяет одинаково обрабатывать FAQ, запись, оплату, эскалацию и фоновые процессы, а затем разбирать сбои по журналу, а не по догадкам.

## 16. Визуальные схемы

Нужно подготовить 2 варианта схемы:

Для Excalidraw используется скилл `excalidraw-architecture` из урока.

Единая техническая схема должна показать:
- 4 канала;
- адаптеры каналов;

- буфер 5 секунд;
- input validation;
- privacy gate и согласия;
- guardrails;
- маскирование ПД;
- context builder;
- Redis;
- PostgreSQL;
- роутер;
- сценарии;
- YCLIENTS;
- ЮKassa;
- scheduler;
- task queue/workers;
- админку;
- эскалацию в Telegram-бот сотрудников;
- monitoring & alerts;
- evals.
- compact длинной истории;
- ограниченный цикл output validation;
