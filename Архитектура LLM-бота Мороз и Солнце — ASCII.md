# Архитектура LLM-бота «Мороз и Солнце» — ASCII-схема

Источник: `Архитектура LLM-бота Мороз и Солнце.excalidraw`.

Схема ниже сохраняет блоки, связи, развилки и последовательности целевой архитектуры. Стрелка `-->` означает переход или передачу данных; `+-->` показывает ветвление; пунктирная смысловая связь подписана словами.

## 2. Развилки роутера и сценарии

Роутер может запустить одну или несколько веток одновременно.

```text
                              / РОУТЕР НАМЕРЕНИЙ \
                                       |
        +---------------+--------------+--------------+---------------+
        |               |              |              |               |
        v               v              v              v               v
  [FAQ / подбор]   [Запись]       [Коммерция]    [Эскалация]     [Scheduler]
        |          перенос /            |          сложный              |
        |          отмена               |          запрос               |
        v               |              v              |               v
  [База знаний]         v       [Резюме покупки] [Создать задачу] [Напоминания]
  цены / правила / [Сбор данных]   и правила            |         post-visit 1–5
  услуги           услуга, время,       |               v         низкая оценка
        |          имя, телефон         v       [Telegram-бот]    -> администратор
        v               |         [ЮKassa: ссылка]  сотрудников          |
  [Основная LLM]        v              |                               v
  безопасный ответ [YCLIENTS]           +--> [Webhook оплаты]    [Task Queue
                   слоты / запись       |    статус в БД          + Workers]
                        |               |
                        +--> [Подтверждение перед действием]
```

От роутера также запускается ветка рассылок:

```text
[Рассылки из админки]
          |
          v
[Черновик / предпросмотр / тест]
          |
          v
[Лимиты / стоп-кнопка / согласие на рекламу / без дублей / отчеты]
          |
          +--> [Реактивация]
               рекламное согласие / idempotency / лимит попыток / «стоп»
               ответ пользователя --> новый диалог
          |
          +--> [Post-visit feedback]
               один запрос на визит
               низкая оценка --> задача администратору
               агрегаты по услуге и специалисту
```

## 3. Внешние системы и фоновые исполнители

```text
[YCLIENTS: слоты и запись] ------ API ------> [YCLIENTS API]
                                              источник слотов и записей

[ЮKassa: платежная ссылка] ----- оплата ----> [ЮKassa API]
                                              платежи / webhooks

[Telegram-бот сотрудников] ---- уведомить --> [Telegram-бот администраторов]

[Task Queue + Workers] --------- задачи ----> [Очередь задач]
                                              ретраи / фоновые работы
```

## 4. Общий выход всех сценариев

FAQ, запись, коммерция, эскалация, scheduler и рассылки сходятся в единый контракт результата.

```text
[FAQ] -----------+
[Запись] --------+
[Коммерция] -----+
[Эскалация] -----+----> +--------------------------------------+
[Scheduler] -----+      | Единый результат сценария            |
[Рассылки] ------+      | status / message / next_action        |
                        | events / cost / error_code            |
                        +-------------------+------------------+
                                            |
                                            v
                        +--------------------------------------+
                        | Output security + validator          |
                        | утечки / ПД / факты / формат         |
                        +-------------------+------------------+
                                            |
                             +--------------+--------------+
                             |                             |
                        ответ прошёл                  ошибка / не прошёл
                             |                             |
                             v                             v
              +-----------------------------+   +---------------------------+
              | Восстановить разрешённые ПД |   | Retry / fallback          |
              | только из текущей сессии    |   | резервная LLM             |
              | ответ в нужный канал        |   | эскалация                 |
              +-------------+---------------+   +-------------+-------------+
                            |                                 |
                +-----------+-----------+                     |
                |                       |                     |
                v                       v                     v
       [Журнал событий]       [Monitoring & Alerts] <------ [алерт]
       correlation_id         здоровье системы
       этапы / ошибки
       latency / токены
       стоимость / KPI
```

Ограничение проверки выхода: если ответ не прошёл validator, допускается максимум один повтор; дальше используется fallback или эскалация.

## 5. Хранилища, управление качеством и админка

```text
                         +----------------------+
                         | Сбор контекста       |
                         +----------+-----------+
                                    |
                  +-----------------+-----------------+
                  |                                   |
                  v                                   v
  +-------------------------------+   +--------------------------------+
  | Redis                         |   | PostgreSQL                     |
  | контекст / буфер / флаги /    |   | история / клиенты / записи /   |
  | кэш                           |   | оплаты / знания / аудит         |
  +-------------------------------+   +--------------------------------+

  +--------------------------------+       +----------------------------+
  | Админка владельца              |<----->| Evals                      |
  | знания / цены / рассылки / KPI |       | роутер / ответы /          |
  | здоровье системы               |       | guardrails / сценарии      |
  +---------------+----------------+       +----------------------------+
                  |
                  +--> управляет знаниями и ценами
                  +--> запускает и контролирует рассылки
                  +--> показывает KPI и здоровье системы
```

## 6. Полная последовательность одного сообщения

```text
Канал
  --> адаптер канала
  --> единый IncomingMessage
  --> нормализация
  --> [если аудио] SpeechKit --> транскрипт
  --> privacy gate
      +--> [нет согласия] обработка без истории и без LLM
      +--> [есть согласие] продолжить
  --> буфер 5 секунд
  --> scripts-first валидация
  --> input guardrails и маскирование ПД
  --> контекст из Redis и базы знаний
  --> [если история > 30 сообщений] summary + последние реплики
  --> роутер намерений
  --> один или несколько сценариев
  --> внешние API / очередь / администратор при необходимости
  --> единый результат сценария
  --> output security + validator
      +--> [ошибка] один retry максимум --> fallback / резервная LLM / эскалация
      +--> [успех] восстановить разрешённые ПД из текущей сессии
  --> ответ в исходный канал
  --> журнал событий
  --> monitoring, alerts и KPI
```
