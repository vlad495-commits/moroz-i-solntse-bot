# Moroz i Solntse Bot

Telegram LLM-assistant for the Moroz i Solntse tanning and cryotherapy center. Step 1 prototype: text questions, LLM answers, Postgres history, Redis context, no booking integrations yet.

> 🟢 **Текущая ступень: 1 — Прототип.** Это минимальный жизнеспособный бот: принимает сообщение → отвечает через LLM → сохраняет историю. Без админки, защиты, эвалов и буфера.
>
> Дальше — апгрейд по командам скилла `llm-project-steps`. См. раздел «Лестница ступеней» внизу.

## Технический стек

- **Язык:** Python 3.12
- **Telegram:** aiogram 3.x
- **LLM:** любой (OpenAI / Anthropic / OpenRouter / DeepSeek / локальный Ollama / ...)
- **Хранилища:** Redis (короткая память — контекст диалога), PostgreSQL (история сообщений)
- **Контейнеризация:** Docker + docker-compose (3 контейнера)

## Опорные документы

- `ТЗ и архитектура.md` — главный целевой документ для дальнейшей архитектуры и реализации: содержит требования и встроенную полную ASCII-схему. Новые решения по каналам, записи, оплатам, админке, guardrails, scheduler, worker, рассылкам, KPI и production-вопросам сверять с ним.
- `Дорожная карта.md` — живой план задач и статусов.
- `changelog.md` — обязательный лог всех значимых действий.

    ├── tmp/                 ← расходники, можно чистить
    ├── logs/                ← логи бота
    │
    └── llm/                 ← КОНТЕЙНЕР 1: бот + LLM
        ├── Dockerfile, requirements.txt
        ├── bot.py           ← точка входа, aiogram polling
        ├── handlers.py      ← /start, обработка текста, отказ на нетекст
        ├── llm.py           ← клиент к LLM-провайдеру (универсальный)
        ├── config.py        ← все настройки из .env

**Контейнеры в `docker-compose.yml` (3 шт):**
1. `llm` — бот + LLM (build из `./llm`)
2. `redis` — `redis:7-alpine` (кэш контекста)
3. `postgres` — `postgres:16-alpine` (история сообщений)

## Поток одного сообщения (ступень 1)

```
Пользователь
    │
    ▼
┌─────────────────────────────────────────────────┐
│ КОНТЕЙНЕР llm                                   │
│                                                 │
│  bot.py (long-polling aiogram)                  │
│    │                                            │
│    ▼                                            │
│  handlers.py                                    │
│    │ 1. проверка длины (MAX_INPUT_LENGTH)       │
│    │ 2. save user message ───── Postgres       │
│    │ 3. typing indicator                        │
│    │                                            │
│    ▼                                            │
│  llm.py                                         │
│    │ контекст диалога ──────── Redis: chat:{id} │
│    │ system_prompt из prompts/system.md         │
│    │ ── LLM-провайдер (один вызов)              │
│    │                                            │
│    ▼                                            │
│  send_message(пользователю)                     │
│    │                                            │
│    └─ message → Postgres                        │
└─────────────────────────────────────────────────┘
```

### Что в Redis
- `chat:{chat_id}:messages` — контекст диалога (последние N сообщений)

### Что в Postgres
- `messages` — вся история (user/assistant)

**Срок хранения:** `messages` чистятся раз в сутки фоновой задачей `_cleanup_loop`. По умолчанию хранятся **3 года** (`DATA_RETENTION_DAYS=1095` в `.env`). Чтобы хранить вечно — `DATA_RETENTION_DAYS=0`.

## ЖЕЛЕЗНОЕ ПРАВИЛО: только через Docker

**Проект ВСЕГДА запускается через Docker.** Никогда напрямую `python bot.py`. Даже локально.

```bash
# Локальный запуск:
cd project && docker compose up --build

# Логи:
docker compose logs -f llm

# Перезапуск контейнера (после изменения кода или .env):
docker compose restart llm

# Остановить всё:
docker compose down
```

**Почему:** код полагается на хосты Redis/Postgres из docker-сети (`redis://redis:6379`, `postgres://postgres:5432`). При запуске напрямую — не подключится.

## ЖЕЛЕЗНОЕ ПРАВИЛО: git

Проект под git с момента инициализации. Ветка `main` — основная.

### 1. После каждого логического шага — коммит

**Логический шаг = одна завершённая единица работы**, например:
- Настроили системный промпт (`prompts/system.md`)
- Поменяли текст-заглушку или температуру (`.env`)
- Поправили багу в `handlers.py`
- Сделали апгрейд ступени (это вообще обязательный коммит)

**Не нужно** коммитить каждое сохранение файла. Коммитим, когда **«что-то заработало целиком»** или **«одна осмысленная правка завершена»**.

**Формат коммита:**
```
<тип>: <короткое описание>

Примеры:
  feat: добавлен системный промпт для ассистента поддержки
  fix: handlers — корректная обработка пустого сообщения
  config: temperature 0.3 → 0.7, max_tokens 2000 → 1500
  upgrade: переход на ступень 2 (админка)
```

**Команды:**
```bash
git add -A
git commit -m "тип: описание"
```

### 2. Эксперименты и работа «в сторону» — отдельная ветка

Если пользователь просит сделать что-то **вне основного потока** (эксперимент, рефакторинг ради рефакторинга, проба новой фичи без гарантии что зайдёт, переписать кусок «по-другому» чтобы сравнить) — **предложи завести отдельную ветку**.

**Зачем:** main всегда остаётся рабочим. Эксперимент пошёл криво — просто переключаешься обратно на main, удаляешь ветку, и как будто ничего не было. Зашёл — мержишь обратно.

**Триггеры — что-то "в сторону":**
- «давай попробуем переписать через X»
- «а что если сделать через Y»
- «эксперимент: добавим Z»
- «хочу проверить идею с W»
- любое «давай попробуем» / «а давай по-другому» — где результат **не гарантирован**.

**Что предлагать:**
> «Это похоже на эксперимент — давай заведём отдельную ветку `experiment/<суть>`, чтобы main не сломать. Если зайдёт — смержим. Если нет — просто удалим ветку.»

**Команды:**
```bash
# Создать и переключиться:
git checkout -b experiment/<короткое-имя>

# Поработали, понравилось — мержим в main:
git checkout main
git merge experiment/<имя>
git branch -d experiment/<имя>

# Не понравилось — выкидываем:
git checkout main
git branch -D experiment/<имя>
```

### 3. Никаких пушей на GitHub без явного запроса

Скилл создаёт **только локальный** git. На GitHub ничего не уходит. Чтобы запушить — нужен отдельный явный шаг (создать репо на GitHub, добавить remote, `git push`). Делается через скилл `commit` или вручную.

## Лестница ступеней (как развивается проект)

| № | Ступень | Что добавляется |
|---|---|---|
| 1 | **Прототип** ← ты здесь | Бот + LLM + Postgres + Redis в Docker |
| 2 | Админка | FastAPI: диалоги, статистика, редактор промпта, тумблер бот вкл/выкл |
| 3 | Эвалы | Тест-кейсы, прогоны через LLM-судью, прогресс через SSE |
| 4 | Буфер | Склейка быстрых сообщений в один LLM-запрос |
| 5 | Безопасность LLM | Guardrails (jailbreak/утечка промпта) + PII-маскирование (152-ФЗ при зарубежном LLM) |
| 6 | Надёжность LLM | Резервный провайдер, retry, алерты в Telegram при ошибках |
| 7 | Деплой | Сервер, SSH, бэкапы Postgres, hardening |
| 8 | Production-ready | Финальный аудит, смена паролей, чек-лист сдачи |

**Переход на следующую ступень:** скажи Codex «следующий шаг» / «апгрейд до ступени N» — скилл `llm-project-steps` подкинет новые файлы и допилит существующие.

**Важно:** твой системный промпт, секреты в `.env` и материалы в `data/` при апгрейде **не теряются**.
