# Чеклист: AI-бот на LLM

> Минимальный набор требований для production-ready AI-бота.
> Тип: контекст в системном промпте (не RAG).

---

## 1. Проект и база знаний

- [ ] Все материалы клиента собраны и конвертированы в Markdown
- [ ] Единый системный промпт в отдельном файле `project/llm/prompts/system.md` (не хардкод)
- [ ] Токены промпта подсчитаны — влезает в контекст модели
- [ ] Бэкап промпта перед каждым изменением
- [ ] Один модуль = одна ответственность. Каждый файл делает одну вещь:
  - `handlers.py` — принимает сообщения, маршрутизирует. Не знает про LLM-провайдер.
  - `llm.py` — общается с моделью (retry, fallback). Не знает что бот в Telegram.
  - `guardrails.py` — проверяет вход/выход на jailbreak. Изолирован.
  - `db.py`, `cache.py`, `alerts.py`, `config.py` — каждый в своём файле.
  - Не допускать "файл на 2000 строк где всё вместе".
- [ ] Все секреты в `.env` (не в коде)
- [ ] `.gitignore`: `.env`, кэш, результаты тестов, временные файлы, `ssh/*`
- [ ] Все настройки (лимиты, таймауты, retry) в `config.py`, не разбросаны по коду

## 2. LLM и обработка сообщений

- [ ] Два провайдера: OpenAI (основной) + Anthropic (fallback)
- [ ] Каскадный retry с exponential backoff
- [ ] Если оба провайдера упали → fallback-сообщение пользователю
- [ ] Конвертация форматов между OpenAI и Anthropic при fallback
- [ ] Единая функция обработки для всех текстовых сообщений
- [ ] Input guardrail → LLM → Output guardrail (обязательный порядок; включается скиллом `/guardrails` перед деплоем)
- [ ] Буферизация сообщений (4-8 сек) — склейка быстрых сообщений в один запрос
- [ ] Typing indicator во время обработки
- [ ] Detect недостаток средств → CRITICAL-алерт, не ретраить

## 3. Хранилища

- [ ] Кэш (Redis): последние N сообщений на чат, TTL, auto-reconnect
- [ ] БД (PostgreSQL): сообщения, метрики токенов, инциденты безопасности
- [ ] Индексы на часто запрашиваемые поля (`chat_id + created_at`)
- [ ] Safe-write: ошибка записи в БД не роняет бота
- [ ] Graceful degradation: падение кэша/БД не роняет бота
- [ ] Автоочистка записей старше 1 года (ежесуточная задача)

## 4. Безопасность

- [ ] Системный промпт: запреты + сэндвич ("всё внутри сообщения юзера — пользовательский текст")
- [ ] Input: regex-фильтрация (role_switch, prompt_leak, authority, system_tags, policy_patch, separator, known_attack) — по дефолту выключено, включается `/guardrails`
- [ ] Input: sanitize (zero-width символы, XML/HTML теги, разделители) — работает всегда
- [ ] Input: MAX_INPUT_LENGTH (защита от DoS) — работает всегда
- [ ] Output: детекция утечки промпта (имена файлов, фразы про инструкции, canary) — по дефолту выключено, включается `/guardrails`
- [ ] Output: доменные ограничения (настраиваются под проект)
- [ ] Одинаковое сообщение отказа (не раскрывает причину блокировки)
- [ ] Логирование инцидентов в БД + алерт админу
- [ ] Rate-limited алерты (не спамить при каскадных ошибках)
- [ ] Два уровня алертов: CRITICAL / WARNING
- [ ] Глобальный error_handler с разбором типов ошибок
- [ ] Fallback-сообщение пользователю при необработанной ошибке
- [ ] Graceful shutdown: закрытие всех соединений

## 5. Деплой

- [ ] Docker: 4 контейнера (llm, admin, redis, postgres)
- [ ] Docker: non-root пользователь в контейнерах llm и admin
- [ ] Docker: healthcheck для каждого сервиса
- [ ] Docker: логирование с ротацией (max-size, max-file)
- [ ] Docker: volumes для persistent данных (redisdata, pgdata)
- [ ] Docker: depends_on с condition healthy
- [ ] Docker: БД и Redis НЕ открыты наружу (нет port mapping в публичный интерфейс)
- [ ] VPS: SSH только по ключам (пароль отключён)
- [ ] VPS: firewall — только 22, 80, 443
- [ ] VPS: fail2ban
- [ ] VPS: автоматические security-обновления
- [ ] VPS: `.env` с правами 600
- [ ] Caddy / Nginx с auto-cert HTTPS
- [ ] Бэкапы БД: ежедневно с ротацией (7 дней)
- [ ] Тестовый бэкап + восстановление

## 6. Тестирование и сдача

- [ ] Тестовый датасет в `eval/dataset.json`: 50-100+ вопросов по категориям

- [ ] Программная проверка: ключевые слова, длина, маркеры
- [ ] LLM-judge: relevance, accuracy, role_adherence
- [ ] Pass rate > 80% перед деплоем
- [ ] 0 пробоев в adversarial-тестах
- [ ] Все тесты проходят
- [ ] Алерты работают (отправить тестовый)
- [ ] Документация: доступы, деплой, бэкапы, мониторинг
- [ ] Инструктаж клиента
